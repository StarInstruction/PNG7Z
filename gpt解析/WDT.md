下面我把这份 **AURIX TC3xx / TriCore 的 WDT（Watchdog Timer）文档**，按“从概念到机制，再到寄存器和实际理解”的顺序，给你系统讲一遍。你把它理解成：**这不只是一个“喂狗定时器”，而是一套“看门狗 + 关键寄存器写保护 + 时序/序列校验”的安全机制**。文档核心内容见第1–3页、第5–12页和后面的寄存器说明。

------

## 1. 先建立整体认识：TriCore 里的 WDT 到底在保护什么

这份文档讲的不是单纯“定期清零防死机”的传统 WDT，而是 AURIX TC3xx 里的 **系统级保护机制**。它主要做三件事：

1. **监控 CPU / 系统是否还在按预期运行**
2. **保护关键寄存器，防止运行期被误写**
3. **通过密码、时间窗口、访问顺序检查，判断软件是否跑飞或异常**

文档一开始就说明：TC3xx 里有 **1 个 Safety Watchdog** 和 **每个 CPU 一个 CPU Watchdog**。每个 WDT 都带有：

- 可编程时基和重装值
- 密码保护
- 时间戳/窗口检查
- 错误刷新会触发 Safety Alarm
- 还带有对关键寄存器的 ENDINIT 写保护机制

------

## 2. WDT 的种类：Safety WDT 和 CPU WDT 有什么区别

### 2.1 Safety Watchdog

Safety WDT 是**系统级**的看门狗，独立于各个 CPU WDT。它主要面向整个系统安全状态，尤其适合监控安全相关的软件执行是否正常。它还能保护一类“**安全关键寄存器**”，这些寄存器通常标成 **SE**。当 Safety ENDINIT 被清除时，才能临时写这些寄存器。第4页明确说明，Safety WDT 的配置、使能/禁止需要 Safety ENDINIT 密码，而且通常只有被授权的“安全 CPU”能操作它。

### 2.2 CPU Watchdog

每个 CPU 都有自己的 WDT，用来监控**该 CPU 自己的执行流**。这比所有核共用一个 WDT 更合理，因为各核不需要协调“谁去喂狗”。文档第4页特别指出：

- **CPU0 在复位后自动运行，CPU0 的 WDT 自动启动**
- 其他 CPU 初始处于 HALT，因此它们的 WDT 默认是禁用的
- 一个 CPU 的 WDT **只能由它自己配置、使能、禁用**
- 标记为 **CEy** 的寄存器，只有对应 CPU 的 ENDINIT 清零时才可写
- 标记为 **E** 的系统关键寄存器，只要任意一个 CPU 的 ENDINIT=0 就可写

这个设计说明：
**CPU WDT 负责“核内执行健康”，Safety WDT 负责“系统整体安全”和更高等级保护”。**

------

## 3. 这套 WDT 最重要的特点：它不是“写个 key 就完事”

第5页总结了 WDT 的核心特性。最值得你抓住的是这几条：

- **16 位计数器**
- 时钟可选：`fSPB/64`、`fSPB/256`、`fSPB/16384`
- 正常模式下用可编程 `REL` 作为重装值
- 带 `ENDINIT` 位
- 对 `WDTxCON0` 的访问采用**复杂密码机制**
- 非法密码或保护位错误，直接上报 **SMU alarm**
- 可选：
  - **代码执行序列检查**
  - **代码执行时间检查**
- 计数器溢出会报错
- 即使禁用 WDT，**访问保护和 ENDINIT 监控仍然保留**

这意味着：
AURIX 的 WDT 更像是一个“**带状态机的安全门卫**”，不是普通 MCU 上那种只要写个寄存器就能一直喂的简单看门狗。

------

## 4. ENDINIT：为什么 WDT 文档里一直在讲它

### 4.1 ENDINIT 是什么

ENDINIT 可以理解为一种“**初始化完成后就锁死关键寄存器**”的机制。第5–6页说得很清楚：很多寄存器只应该在初始化阶段配置一次，运行时改它们可能导致系统失控。于是 AURIX 用 ENDINIT 来做额外保护。

简单说：

- `ENDINIT = 1`：关键寄存器**禁止写**
- `ENDINIT = 0`：在**主管态（Supervisor Mode）**下，关键寄存器允许写

但它并不是“随便清零再写”。
因为：

1. 清 ENDINIT 必须走严格的密码访问流程
2. 一旦清掉，WDT 会开始一个**超时窗口**
3. 你必须在窗口内把 ENDINIT 再置回 1
4. 如果超时没锁回去，系统认为你软件异常，触发故障响应

### 4.2 哪几类寄存器受保护

文档第5–6页定义了三类：

- **CEy**：只能在 `CPUy WDT ENDINIT = 0` 时写
- **E**：系统关键寄存器，只要任一 CPU WDT 的 ENDINIT=0，或者 `EICON0.ENDINIT=0` 就可写
- **SE**：安全关键寄存器，只能在 `Safety WDT ENDINIT=0` 或 `SEICON0.ENDINIT=0` 时写

这是理解 AURIX 启动代码、时钟配置、复位配置、保护寄存器修改流程的关键。

## 6. 访问 WDT 控制寄存器的三种方式：这是全文最核心的机制

WDT 的关键寄存器是 `WDTxCON0`。访问它不是普通写寄存器，而是有三种操作：

### 6.1 Password Access

这是“**第一步，解锁**”。

你必须往 `WDTxCON0` 写入一个符合规则的值，硬件检查密码和附加字段是否正确。成功后：

- `LCK` 会被清零，表示寄存器解锁
- WDT 进入 **Time-Out Mode**
- 接下来你才允许做 Modify Access

如果密码不对：

- `WDTxSR.AE = 1`
- 发送 SMU alarm

### 6.2 Check Access

它和 Password Access 很像，但**不会真正解锁**。
用途是：在正常服务之间做“中间检查点”，比如检查代码执行顺序、执行时间是不是符合预期。若写入不符合要求，也会报 Access Error。第8–9页强调了这一点。

### 6.3 Modify Access

这是“**第二步，真正修改**”。

只有在 Password Access 成功解锁后，下一次写 `WDTxCON0` 才会被当作 Modify Access。此时你可以：

- 设置 `ENDINIT`
- 更新密码 `PW`
- 更新重装值 `REL`

完成后硬件会自动：

- 把 `LCK` 重新置 1
- 再次锁住寄存器

------

## 7. 为什么密码访问这么复杂

文档第6–9页实际上在告诉你：
**它要防止“软件跑飞后误打误撞写对寄存器”**。

### 7.1 静态密码模式

如果 `PAS=0`，密码不会自动变。
但读取 `PW` 时，低一部分位会被“翻转返回”，因此你**不能简单读出来再原样写回去**。这是故意设计来防止简单故障路径的。

### 7.2 自动密码序列模式

如果 `PAS=1`，每次 Password/Check 后，密码会按一个 **14 位 Fibonacci LFSR** 序列自动前进。多项式是：
`x14 + x13 + x12 + x2 + 1`。
图72展示了这个 LFSR。这样做的目的是让软件必须按预定序列运行，不能随便重复旧密码。

### 7.3 时间检查密码

如果 `TCS=1`，Password Access 或 Check Access 时，`REL` 字段不能简单写原来的 `REL`，而要写**当前计数值 TIM 的按位取反估计值**，并且误差必须落在 `±TCT` 范围内。
这其实就是：

> “你不仅得知道密码，还得知道现在大概过去了多久。”

这能检测：

- 代码有没有提前跑到这里
- 有没有卡太久
- 任务顺序/时间是否异常

------

## 8. WDT 的三种工作模式

第10–11页把 WDT 工作模式说得非常清楚：

### 8.1 Time-Out Mode

进入方式：

- Application Reset 后，CPU0 WDT 默认进入这个模式
- 执行一次有效的 Password Access 后，也会进入这个模式

此时：

- 计数器从固定值 `0xFFFC` 开始往上数
- 必须在超时前完成后续 Modify Access，把 ENDINIT 正确设回去
- 否则报错并通知 SMU

你可以把它理解成：
**这是“解锁后办事”的临时窗口模式。**

### 8.2 Normal Mode

这是普通“看门狗工作模式”：

- 计数器从 `REL` 开始计数
- 你要在溢出前完成一次合法服务
- 否则触发 WDT 超时告警

### 8.3 Disabled Mode

CPU0 之外的 CPU WDT 在应用复位后默认一般是这个模式。
注意：**Disabled 不是完全没保护了**。文档特别强调，禁用的只是“常规 watchdog 功能”，但：

- 如果你去访问 `WDTxCON0`
- 仍然会进入 Time-Out Mode
- 非法访问仍然会触发告警

也就是：
**禁用 WDT != 关闭保护逻辑**。

------

## 9. “喂狗”到底怎么做

第11页给出了标准服务流程。你要记住：

### 标准服务 = 两步

1. **有效 Password Access**
2. **紧接着有效 Modify Access**

其中第 1 步会自动把 WDT 切到 Time-Out Mode，所以第 2 步必须及时完成。
而且在第 2 步中，**`ENDINIT` 必须写 1**，哪怕它本来已经是 1。文档专门强调了这一点。

这句话很重要，因为很多人会误以为：

> “我不改 ENDINIT，只是喂狗，那 ENDINIT 保持原值就行。”

不行。文档明确说：**正常服务时也必须把 ENDINIT 写成 1。**

------

## 10. 什么时候会触发告警

第11页总结了三种主要触发场景：

1. **访问数据非法**
   - 包括 `WDTxCON0`、`EICON0`、`SEICON0` 写入格式不对
2. **在 Time-Out Mode 下超时**
   - 比如做了 Password Access，但没及时完成 Modify Access
3. **在 Normal Mode 下超时**
   - 也就是该喂狗时没喂

而最终怎么响应，不是 WDT 自己决定，而是 **SMU 配置**决定，可以是 trap、interrupt、reset 等。第3页和第11页都讲到了。

## 11. 定时周期怎么算

第10页给出通式：

[
\text{period} = \frac{(2^{16} - startvalue)\times divider}{f_{SPB}}
]

其中：

- `startvalue`
  - 正常模式：`REL`
  - Time-Out 模式：固定 `0xFFFC`
- `divider` 由 `IR1:IR0` 决定：
  - `00 -> fSPB/16384`
  - `01 -> fSPB/256`
  - `10 -> fSPB/64`
  - `11 -> 保留，不能用`

所以 Time-Out Mode 的窗口通常很短，因为它从 `0xFFFC` 起数，只差 4 个计数就溢出了；这就是它被设计成一个“短暂开窗”的原因。正常模式窗口长短则主要由 `REL` 和分频决定。

## 12. EICON / SEICON：为什么还要有“全局 ENDINIT 控制寄存器”

这是很多初学者容易忽略，但实际上非常实用的点。

### 12.1 为什么需要 EICON / SEICON

文档第9页就提到，有些场景不希望通过 WDT 去开窗：

- WDT 本身可能被禁用
- 你不想打乱 WDT 的时间检查逻辑
- 你只想临时改一下受 ENDINIT 保护的寄存器，而不影响正常喂狗节奏

于是有了：

- `EICON0/EICON1/EISR`：系统 ENDINIT 的独立超时控制
- `SEICON0/SEICON1/SEISR`：Safety ENDINIT 的独立超时控制

### 12.2 它们和 WDT 的区别

EICON/SEICON 的特点（第26–33页）：

- 也要密码访问
- 也有 timeout 机制
- 但它们的 `REL` 是固定的 `0xFFFC`
- 密码是**静态**的，不支持自动序列
- 更适合“只为了开/关 ENDINIT 窗口”而设计

你可以把它理解成：

- **WDTxCON0**：既管喂狗，又管 ENDINIT
- **EICON/SEICON**：只管 ENDINIT 开窗，不干扰 CPU WDT 的正常服务节奏

## 13. 关键寄存器应该怎么读

下面给你一个“读寄存器抓重点”的方式。

### 13.1 `WDTxCON0`

核心字段：

- `ENDINIT`：是否允许写受保护寄存器
- `LCK`：是否锁住
- `PW`：密码
- `REL`：重装值 / 时间检查字段

### 13.2 `WDTxCON1`

核心字段：

- `DR`：请求使能/禁用 WDT
- `IR1:IR0`：时钟分频
- `UR`：是否限制在 SMU 非 FAULT 状态才能解锁
- `PAR`：是否启用自动密码序列
- `TCR`：是否启用计数时间检查
- `TCTR`：时间检查容差

### 13.3 `WDTxSR`

最关键的状态位：

- `AE`：访问错误
- `OE`：计数器溢出错误
- `IS1:IS0`：当前输入分频
- `DS`：是否禁用
- `TO`：是否处于 Time-Out 模式
- `US`：SMU 解锁限制状态
- `PAS`：是否启用自动密码序列
- `TCS`：是否启用时间检查
- `TCT`：容差
- `TIM`：当前计数值

## 14. 学这篇文档时，最容易混淆的几个点

### 误区 1：WDT 只是“防死循环”

不对。
这份文档表明它还在做：

- 密码保护
- ENDINIT 写保护
- 序列检查
- 时间检查
- 访问合法性判断

### 误区 2：禁用 WDT 就没有任何保护了

也不对。
禁用的是常规计时看门狗功能，但 WDT 控制寄存器的访问保护、ENDINIT 监控仍然存在。

### 误区 3：清 ENDINIT 后可以马上写关键寄存器

文档第6页专门提醒：
**清除 ENDINIT 需要时间生效，最好读回确认 ENDINIT 已真的清零后，再访问受保护寄存器。**

### 误区 4：喂狗时 ENDINIT 不用管

不对。
有效服务要求在 Modify Access 中把 `ENDINIT` 写成 1。

### 误区 5：EICON/SEICON 是多余的

恰恰不是。
它们是为了让你能在**不影响 CPU WDT 正常运行**的前提下，独立开一个 ENDINIT 写窗口。

## 15. 你可以把整套机制想象成下面这个流程

### 正常运行时

- WDT 在 Normal Mode
- 从 `REL` 开始计数
- 软件周期性执行合法服务

### 要修改关键寄存器时

1. 先做合法 Password Access
2. WDT 进入 Time-Out Mode
3. 再做合法 Modify Access，把 `ENDINIT=0`
4. 读回确认 ENDINIT 已清零
5. 写受保护寄存器
6. 再做一次合法解锁/修改，把 `ENDINIT=1`
7. WDT 回到正常状态

如果你不想影响 WDT 正常节奏，就用 `EICON` / `SEICON` 来完成中间这件事。

------

## 16. 这份文档最值得你背下来的 8 个结论

1. **CPU0 的 WDT 在应用复位后自动启动；其他 CPU WDT 默认不启动。**
2. **WDT 不仅看门狗，还负责 ENDINIT 保护。**
3. **访问 `WDTxCON0` 必须按 Password Access → Modify Access 的两步流程。**
4. **Password Access 成功后会进入 Time-Out Mode，窗口很短。**
5. **非法密码/非法格式/超时都会触发 SMU Alarm。**
6. **可启用自动密码序列（LFSR）和计数时间检查。**
7. **系统级 ENDINIT 和 Safety ENDINIT 可以通过 EICON/SEICON 单独控制。**
8. **禁用 WDT 不等于禁用访问保护。**
